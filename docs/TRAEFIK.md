# Traefik Setup and Networking Model

This document describes how Traefik is configured as the reverse proxy for all services managed by `sudobility_dockerized`, including networking, service discovery, SSL certificate provisioning, and troubleshooting.

## Architecture Overview

```
        Internet
           |
     ┌─────┴─────┐
     │  Port 80   │  (HTTP - ACME challenges + redirect)
     │  Port 443  │  (HTTPS - all service traffic)
     └─────┬─────┘
           │
    ┌──────┴──────┐
    │   Traefik   │  (Reverse Proxy)
    │  container  │
    └──────┬──────┘
           │
    sudobility_network (Docker bridge)
           │
     ┌─────┼──────────┐
     │     │          │
  ┌──┴──┐ ┌┴────┐ ┌──┴──┐
  │svc_a│ │svc_b│ │svc_c│   (Backend service containers)
  └─────┘ └─────┘ └─────┘
```

All traffic enters through Traefik on ports 80 and 443. Traefik routes requests to the correct backend container based on the `Host` header (hostname-based routing). Each service runs on an internal port that is never exposed to the host.

## The `sudobility_network` Docker Network

### How It Is Created

The shared Docker bridge network `sudobility_network` is created in two places for redundancy:

1. **`setup-scripts/common.sh` > `ensure_network()`** -- Pre-creates the network before Traefik starts. This function is called by `install_traefik()` and is idempotent (safe to call multiple times).

2. **Traefik's `docker-compose.yml`** -- Declares the network with `name: sudobility_network` and `driver: bridge`. Docker Compose will create it if it does not already exist.

### How Services Join the Network

Each service's `docker-compose.yml` (generated by `add.sh`) references the network as external:

```yaml
networks:
  sudobility_network:
    external: true
```

This means the network must exist before the service starts. The `add.sh` script ensures this by calling `install_traefik` (which calls `ensure_network`) as its first step.

### Why a Shared Network?

Docker containers on the same bridge network can communicate by container name. Traefik needs network access to backend containers to forward HTTP requests. Without a shared network, Traefik cannot reach the services.

## Service Discovery via Docker Labels

Traefik is configured with `--providers.docker=true` and `--providers.docker.exposedbydefault=false`. This means:

- Traefik watches the Docker socket (`/var/run/docker.sock`) for container events.
- Only containers with the label `traefik.enable=true` are registered as backends.
- No manual configuration file updates are needed when adding or removing services.

### Labels Applied to Each Service

When `add.sh` creates a service, it generates these Docker labels:

```yaml
labels:
  # Tell Traefik to route traffic to this container
  - "traefik.enable=true"

  # Route requests where Host header matches the configured hostname
  - "traefik.http.routers.<service_name>.rule=Host(`<hostname>`)"

  # Only accept HTTPS connections (port 443)
  - "traefik.http.routers.<service_name>.entrypoints=websecure"

  # Use the "letsencrypt" certificate resolver for automatic TLS
  - "traefik.http.routers.<service_name>.tls.certresolver=letsencrypt"

  # Forward traffic to this port inside the container
  - "traefik.http.services.<service_name>.loadbalancer.server.port=<port>"
```

Each service has a unique router name (`<service_name>`) which prevents routing conflicts between services.

## SSL Certificate Provisioning

### How It Works

Traefik uses **Let's Encrypt** with the **ACME HTTP-01 challenge** to automatically provision SSL certificates:

1. When a new service is added with a hostname (e.g., `api.example.com`), Traefik detects the router labels and sees it needs a certificate.
2. Traefik requests a certificate from Let's Encrypt for that hostname.
3. Let's Encrypt sends an HTTP request to `http://<hostname>/.well-known/acme-challenge/<token>` on port 80.
4. Traefik responds with the expected token (via the `web` entrypoint on port 80).
5. Let's Encrypt validates domain ownership and issues the certificate.
6. Traefik stores the certificate in `/data/acme.json` (persisted in the `sudobility_traefik_data` Docker volume).
7. Certificates are automatically renewed before expiry.

### Configuration in Traefik

```
--certificatesresolvers.letsencrypt.acme.httpchallenge=true
--certificatesresolvers.letsencrypt.acme.httpchallenge.entrypoint=web
--certificatesresolvers.letsencrypt.acme.email=admin@sudobility.com
--certificatesresolvers.letsencrypt.acme.storage=/data/acme.json
```

### HTTP to HTTPS Redirect

All HTTP traffic on port 80 is automatically redirected to HTTPS on port 443:

```
--entrypoints.web.http.redirections.entrypoint.to=websecure
--entrypoints.web.http.redirections.entrypoint.scheme=https
```

The only exception is ACME challenge requests, which Traefik handles before the redirect.

## Setup Script Relationships

```
add.sh
  └── sources setup-scripts/common.sh
        ├── ensure_network()     -- Creates sudobility_network if missing
        ├── install_traefik()    -- Generates Traefik docker-compose.yml and starts it
        ├── ensure_dirs()        -- Creates config-generated/ directory structure
        ├── validate_doppler_token() -- Validates Doppler service token
        ├── fetch_doppler_secrets()  -- Downloads .env from Doppler
        ├── save_doppler_token()     -- Persists token for upgrade operations
        ├── service_pull()       -- Pulls latest Docker image
        ├── service_start()      -- Starts the service container
        └── service_wait_and_verify() -- Checks container is running

setup-scripts/traefik.sh    -- Additional Traefik helpers (hostname prompts, ACME email, dynamic config)
setup-scripts/doppler.sh    -- Doppler secrets management (token prompts, secret fetching, env merging)
setup-scripts/deps_setup.sh -- System dependency installation (Docker, curl, jq, crontab)
```

`common.sh` is the primary shared library. The other setup scripts (`traefik.sh`, `doppler.sh`, `deps_setup.sh`) contain supplementary functions and are not sourced by the main scripts (`add.sh`, `upgrade.sh`, `remove.sh`, `status.sh`) directly.

## Troubleshooting

### DNS Issues

**Symptom:** Certificate provisioning fails; Traefik logs show ACME errors.

**Cause:** The hostname's DNS record does not point to the server's IP address.

**Fix:**
1. Verify DNS resolution: `dig +short <hostname>` or `nslookup <hostname>`
2. Ensure the returned IP matches your server's public IP.
3. DNS changes can take up to 48 hours to propagate, but typically complete within minutes.
4. After fixing DNS, restart Traefik to retry certificate provisioning:
   ```bash
   cd config-generated/traefik && docker compose restart
   ```

### Port 80 Access

**Symptom:** ACME HTTP-01 challenge fails; certificates are not issued.

**Cause:** Port 80 is blocked by a firewall or another process.

**Fix:**
1. Check if port 80 is open: `curl -I http://<server-ip>/`
2. Check firewall rules:
   - Ubuntu/Debian: `sudo ufw status`
   - CentOS/RHEL: `sudo firewall-cmd --list-all`
   - Cloud providers: check security group / firewall rules in the console
3. Check if another process is using port 80: `sudo lsof -i :80`
4. Ensure both ports 80 and 443 are allowed inbound in your firewall.

### Certificate Issues

**Symptom:** Browser shows "insecure" warning or certificate error.

**Possible causes and fixes:**

1. **Certificate not yet issued** -- New certificates can take 1-2 minutes. Check Traefik logs:
   ```bash
   cd config-generated/traefik && docker compose logs -f
   ```
   Look for "Obtaining certificate" or error messages from the ACME provider.

2. **Rate limiting** -- Let's Encrypt has [rate limits](https://letsencrypt.org/docs/rate-limits/) (50 certificates per registered domain per week). If you hit the limit, wait or use a different subdomain.

3. **Corrupted ACME storage** -- If `acme.json` is corrupted:
   ```bash
   # Stop Traefik
   cd config-generated/traefik && docker compose down
   # Remove the ACME volume (certificates will be re-provisioned)
   docker volume rm sudobility_traefik_data
   # Restart Traefik
   docker compose up -d
   ```

4. **Wrong email** -- The ACME email is set in the Traefik docker-compose.yml. To change it, edit `config-generated/traefik/docker-compose.yml` and restart Traefik.

### Service Not Reachable

**Symptom:** `503 Service Unavailable` or `404` when accessing the hostname.

**Fix:**
1. Check the service is running: `./status.sh`
2. Check the service is on the correct network:
   ```bash
   docker inspect <service_name> --format '{{json .NetworkSettings.Networks}}' | jq .
   ```
   It should include `sudobility_network`.
3. Check Traefik can see the service's labels:
   ```bash
   docker inspect <service_name> --format '{{json .Config.Labels}}' | jq .
   ```
4. Check Traefik logs for routing errors:
   ```bash
   cd config-generated/traefik && docker compose logs --tail 50
   ```
5. Verify the service's internal PORT matches the Traefik label:
   ```bash
   grep PORT config-generated/services/<service_name>/.service.conf
   ```

### Traefik Not Starting

**Symptom:** `./status.sh` shows Traefik as "not running".

**Fix:**
1. Check Traefik logs:
   ```bash
   cd config-generated/traefik && docker compose logs
   ```
2. Common causes:
   - Port 80 or 443 already in use by another process
   - Docker socket permissions (Traefik needs read access to `/var/run/docker.sock`)
   - Docker daemon not running
3. Restart Traefik:
   ```bash
   cd config-generated/traefik && docker compose down && docker compose up -d
   ```
